## Scope-cut — что выкидывать, если не укладываюсь (в порядке приоритета)

1. **FastAPI-эндпоинт** → остаётся только CLI. Retrieval + генерация важнее второго интерфейса (−1 ч).
2. **Автоматический подсчёт hit-rate** → 10 вопросов прогоняю руками, оценки в Markdown-таблицу. Понятие eval сохранено, кода меньше (−1 ч).
3. **Heading-aware chunking** → простое скользящее окно по всему документу без секций. Trade-off всё равно можно обсудить в README (−0.5–1 ч).
4. **save/load индекса** → индекс строится в памяти при каждом запуске (~2–3 мин). Некрасиво, но работает (−0.5 ч).

Что **нельзя** выкидывать: retrieval (CP4), генерацию с источниками (CP5), хотя бы ручную оценку — без них это не RAG-проект.

## Стретч-цели — если осталось время (в порядке ценности)

1. **Реранкинг** — cross-encoder (`cross-encoder/ms-marco-MiniLM-L-6-v2`): достаём top-20, переранжируем в top-5. Заметно улучшает retrieval, +1 файл. Сравни hit-rate до/после — готовый пункт в резюме.
2. **Гибридный поиск** — BM25 (`rank_bm25`) + семантический, слияние результатов (RRF). Показывает понимание пределов dense retrieval.
3. **Стриминг ответа** — `stream=True` у Groq + `StreamingResponse` в FastAPI.
4. **A/B chunking в eval** — прогнать eval при chunk_size 100/200/400 слов, таблицу с hit-rate в README. Дёшево и очень наглядно для собеседования.
