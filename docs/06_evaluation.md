# 06 — Оценка качества

**Время: ~2 часа** · Файлы: `data/eval_questions.json`, `rag/evaluate.py`

## Цель модуля

На выходе — свой eval-сет из 15 вопросов и скрипт, считающий retrieval hit-rate@k, плюс заполненная руками таблица faithfulness/relevance для ответов. Ты поймёшь, что RAG оценивается **в два уровня** — «нашли ли нужное?» и «правильно ли ответили по найденному?» — и что без цифр любые слова «у меня работает» ничего не значат (в том числе в резюме).

## Теория-минимум

Два независимых уровня качества:

1. **Retrieval** — попал ли нужный документ в top-k. Метрика: **hit-rate@k** = доля вопросов, где ожидаемый файл есть среди источников top-k чанков. Дёшево, автоматически, объективно.
2. **Generation** — качество текста ответа. Две стандартные оси:
   - **Faithfulness** — всё ли в ответе подтверждается предоставленным контекстом (ловим галлюцинации);
   - **Answer relevance** — отвечает ли ответ на заданный вопрос (а не на соседний).

Автоматически generation оценивают через LLM-as-judge (одна модель судит другую), но у судьи свои искажения — для первого проекта честнее и полезнее **ручная разметка по шкале 0/1/2**. Ключевой инсайт, который стоит проговорить в README: faithfulness и relevance независимы — ответ может быть на 100% верен контексту, но не отвечать на вопрос, и наоборот.

Углубиться: [документация Ragas — Faithfulness](https://docs.ragas.io/en/stable/concepts/metrics/available_metrics/faithfulness/) (сами понятия, библиотека не нужна).

## Пошаговая инструкция

1. Сначала — вопросы. Пройдись по дереву `docs/en/docs/`, выбери разделы, сформулируй вопросы **как реальный пользователь**, а не копипастой заголовков. Копипаста заголовка — читерство: она матчится тривиально и ничего не проверяет.
2. `run_eval` — тонкий цикл поверх уже готовых `search` и `answer_question`. Нового кода мало.
3. Прогони. Сначала смотри только hit-rate: если < 0.7 — проблема в retrieval, чинить генерацию бессмысленно (мусор на входе → мусор на выходе). Типичные причины: слишком большие чанки, мусор из модуля 01, вопрос-переформулировка не матчится (это нормально для 1–2 штук — так и запиши в выводы).
4. Потом читай ответы и заполняй таблицу faithfulness/relevance. На 15 вопросов уйдёт ~30 минут — это нормальная и важная работа.
5. Итоговые цифры и 2–3 наблюдения (что ломается и почему) — в README. Это самая «дорогая» часть проекта для собеседования.

## Как проверить

Мини-assert на саму логику метрики:

```python
fake = [SearchResult(chunk=Chunk(0, "tutorial/query-params.md", "h", "t"), score=0.9)]
assert retrieval_hit({"expected_doc": "tutorial/query-params.md"}, fake)
assert not retrieval_hit({"expected_doc": "tutorial/cors.md"}, fake)
```

Прогон `python -m rag.evaluate` печатает результат по каждому вопросу и итоговый hit-rate. Ориентиры: hit-rate@5 ≥ 0.8 — хорошо; 0.6–0.8 — приемлемо, опиши почему; обе ловушки дали отказ.
