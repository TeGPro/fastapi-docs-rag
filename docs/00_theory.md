# 00 — Теория-минимум: что такое RAG и зачем он нужен

**Время: ~30 минут** (чтение + настройка окружения). Кода здесь нет.

## Зачем RAG

У LLM три фундаментальные проблемы: (1) знания заморожены на дате обучения, (2) модель не знает твоих приватных/нишевых данных, (3) на вопросы вне своих знаний она склонна **галлюцинировать** — уверенно выдумывать. RAG (Retrieval-Augmented Generation) решает все три одним ходом: перед генерацией мы **находим** релевантные куски из своей базы знаний и **кладём их в промпт**. Модель отвечает не «из головы», а по предоставленному контексту.

Формула: `ответ = LLM(вопрос + найденные_куски_документации)`.

## Как устроен пайплайн

Две независимые фазы:

**Индексация (офлайн, один раз):**
1. *Загрузка* — читаем документы (у нас — Markdown-файлы доков FastAPI).
2. *Chunking* — режем на куски (чанки). Целый документ слишком велик и «размывает» поиск; ищем по кускам.
3. *Эмбеддинги* — каждый чанк превращаем в вектор так, что близкие по смыслу тексты получают близкие векторы.
4. *Векторное хранилище* — складываем векторы в индекс, умеющий быстро искать ближайшие.

**Ответ на вопрос (онлайн, на каждый запрос):**
5. *Retrieval* — эмбеддим вопрос той же моделью, ищем top-k ближайших чанков (semantic search).
6. *Генерация* — собираем промпт: инструкция + найденные чанки + вопрос → LLM пишет ответ.
7. *Оценка* — проверяем, что retrieval находит нужное, а генерация не врёт.

## Ключевые термины

| Термин | Смысл |
|---|---|
| **Chunk** | Кусок документа фиксированного размера; единица поиска |
| **Overlap** | Перекрытие соседних чанков, чтобы мысль не разрезалась на границе |
| **Embedding** | Вектор (у нас 384 числа), кодирующий смысл текста |
| **Cosine similarity** | Мера близости векторов; у нормализованных векторов = скалярное произведение |
| **Top-k retrieval** | Взять k ближайших чанков к вектору вопроса |
| **Context window** | Лимит входа модели. Важно: и у LLM, и у **модели эмбеддингов** (наша MiniLM обрезает всё после 256 токенов — это определит размер чанка!) |
| **Grounding** | Ответ опирается только на предоставленный контекст |
| **Faithfulness** | Метрика: нет ли в ответе того, чего не было в контексте |
| **Relevance** | Метрика: отвечает ли ответ на заданный вопрос |
| **Hit-rate@k** | Доля вопросов, где нужный документ попал в top-k |

## Почему пишем руками, а не LangChain

LangChain прячет каждый из шагов за абстракцией. На собеседовании спросят «какой у тебя chunk size и почему», «какая метрика близости», «что в промпте» — после ручной реализации ты ответишь на всё. Низкоуровневые библиотеки (sentence-transformers, faiss, groq) — это честные инструменты, а не чёрный ящик: они не принимают решений за тебя.
